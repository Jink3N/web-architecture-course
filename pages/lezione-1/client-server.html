🌐 Modello Client-Server

Il modello client-server è il paradigma fondamentale dell'architettura web moderna. Divide chiaramente i ruoli tra chi richiede servizi (client) e chi li fornisce (server). Questa separazione permette scalabilità, riusabilità e gestione centralizzata.

        Client (Browser)          Internet/Network            Server (Backend)
        ┌──────────────┐   HTTP/HTTPS Request (GET/POST)   ┌──────────────────┐
        │ HTML/CSS/JS  │  ───────────────────────────────▶ │  Application      │
        │ Rendering UI │ ◀───────────────────────────────  │  Business Logic   │
        └──────────────┘    HTTP/HTTPS Response (HTML/JSON)└──────────────────┘

🔍 Ruoli Principali

Client

Interfaccia utente (browser, mobile app, IoT) che invia richieste e visualizza risultati. Esegue codice JavaScript, gestisce interazioni, caching locale e rendering grafico.

Responsabilità: UX, validazione base, chiamate API.

Server

Elabora logica applicativa, autentica utenti, accede ai dati, applica regole di business e genera risposte strutturate.

Responsabilità: Sicurezza, business rules, data orchestration.

Database

Persistenza dati consistente e affidabile. Può essere relazionale (SQL) o non-relazionale (NoSQL) in base ai casi d'uso.

Responsabilità: Integrità, performance query, transaction management.

API Layer

Interfaccia di comunicazione strutturata tra client e server. Espone risorse e operazioni tramite endpoint REST, GraphQL o gRPC.

Responsabilità: Contract, versioning, data serialization.

⚙️ Tipi di Interazioni

  • Request/Response: Il client invia una richiesta HTTP, il server risponde con dati o errori
  • Stateless: Ogni richiesta deve contenere tutte le informazioni necessarie (RESTful principle)
  • Session-based: Persistenza stato utente mediante session ID (cookie) in alcuni scenari tradizionali
  • Token-based: JWT o OAuth2 per autenticazione scalabile

📡 Esempio di Richiesta HTTP

GET /api/users/123 HTTP/1.1
        Host: example.com
        Accept: application/json
        Authorization: Bearer <token>

Il server elabora, accede al database e risponde con:

{
          "id": 123,
          "name": "Mario Rossi",
          "role": "admin"
        }

🚀 Evoluzione verso Architetture Moderne

Il modello client-server monolitico si è evoluto in sistemi distribuiti:

  1. Monolite: Un'unica codebase e deploy
  2. N-tier: Separazione presentazione, logica e dati
  3. SOA: Servizi integrati via ESB
  4. Microservizi: Servizi piccoli autonomi distribuiti
  5. Serverless: Funzioni on-demand senza gestione server

✅ Vantaggi del Modello

🔐 Controllo Centralizzato

Logica e sicurezza gestite in un'unica posizione, facilitando audit e aggiornamenti.

📦 Riutilizzo

Servizi riusabili da diversi client (web, mobile, IoT).

⚖️ Scalabilità

Possibilità di scalare componenti backend indipendentemente dal client.

📊 Osservabilità

Monitoraggio centralizzato di traffico, errori, performance.

⚠️ Svantaggi e Limiti

🧱 Single Point of Failure

Se il server cade e non c'è ridondanza, l'intero sistema diventa irraggiungibile.

📉 Scalabilità Limitata

Un monolite massivo ha limiti di scaling rispetto a microservizi specializzati.

🚧 Dipendenze Forti

Client tightly coupled rende più difficile evolvere l'API senza breaking changes.

⏳ Latenza Multipla

Richieste multiple da una pagina possono aumentare tempi di caricamento.

💡 Best Practices Moderne

  • API Versioning: /v1, /v2 per evoluzione sicura
  • Caching Layer: CDN e reverse proxy (es. Nginx, Varnish)
  • Security: Rate limiting, TLS ovunque, header di sicurezza
  • Monitoring: Tracing distribuito (OpenTelemetry), logging strutturato

🔄 Esempio di Flusso

1

User Action

L'utente clicca su "Login" nel browser.

2

HTTP Request

Il client invia POST /api/auth/login con credenziali JSON.

3

Server Processing

Il server valida, verifica DB, genera token JWT.

4

Response

Server restituisce 200 OK con token e profilo utente.

5

State Update

Il client salva token (memory/secure storage) e aggiorna UI.
