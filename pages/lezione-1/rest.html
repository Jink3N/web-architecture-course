🏗️ Architettura REST

REST definisce vincoli architetturali per creare sistemi distribuiti scalabili, cacheable e indipendenti.

📜 Origini

Formalizzata da Roy Fielding nella sua tesi (2000) mentre co-progettava HTTP/1.1 e Apache. REST non è un protocollo ma uno stile architetturale.

🎓 Curiosità: Roy Fielding

Roy Fielding co-creò HTTP/1.1 e Apache HTTP Server mentre definiva i principi REST, basandosi su osservazioni di sistemi resilienti e scalabili del web.

🧩 Sei Vincoli Fondamentali

1. Client-Server

Separazione responsabilità: UI e dati indipendenti → evoluzione separata.

2. Stateless

Ogni richiesta contiene tutto il contesto necessario. Nessuno stato lato server di sessione.

3. Cacheable

Risposte marcate come cacheable riducono latenza e carico.

4. Uniform Interface

Identificazione risorse (URI), manipolazione tramite rappresentazioni, messaggi self-descriptive, HATEOAS.

5. Layered System

Proxy, gateway, CDN trasparenti ai client → scalabilità e sicurezza.

6. Code on Demand (Opz.)

Download di script eseguibili (es. JavaScript) per estendere funzionalità client.

🔗 Uniform Interface Dettagli

Identificazione Risorse

URI stabili

  • /users/123
  • /orders/2024/987
  • /products/{sku}

Rappresentazioni

JSON / XML / HAL

  • Content negotiation
  • Media types specifici
  • Versioning via header

Self-Descriptive Messages

Headers + Body

  • Status codes semantici
  • Cache-Control, ETag
  • Link relations

HATEOAS

Hypermedia as Engine

  • Client scopre azioni
  • Riduce coupling
  • Link navigabili

🧪 Idempotenza & Sicurezza Metodi

GET

Safe + Idempotent

PUT

Idempotent (sostituisce)

DELETE

Idempotent (elimina risorsa)

POST

Non idempotent (crea/azione)

📦 Versioning Strategie

URL Versioning

/v1/users
/v2/users

Semplice ma rompe caching.

Header Version

Accept: application/vnd.app.user+json;version=2

Più pulito, flessibile.

Media Type

application/vnd.company.user-v3+json

Documentato e esplicito.

SemVer nei Link

Link di transizione a nuove versioni.

⚙️ Error Handling Standard

Problem+JSON

{
  "type": "https://api.example.com/errors/invalid-input",
  "title": "Validation Failed",
  "status": 422,
  "detail": "Email non valida",
  "field": "email"
}

Retry-After

HTTP/1.1 429 Too Many Requests
Retry-After: 120

Rate limiting esplicito.

ETag Workflow

GET /resource
ETag: "abc123"

PUT /resource
If-Match: "abc123"

Concorrenza ottimistica.

Paginazione

Link: ; rel="next"

Hypermedia navigabile.

✅ Benefici

Scalabilità

Stateless + cache = throughput elevato.

Indipendenza

Client e server evolvono separati.

Interoperabilità

Standard HTTP e media types.

Visibilità

Proxy e gateway ispezionano richieste facilmente.

❌ Limiti

Over/Under Fetching

Richieste multiple o dati eccessivi (GraphQL risolve).

No Transazioni Distribuite

Manca supporto nativo a processi multi-step ACID.

Versioning Complesso

Gestione evoluzione contratti non standardizzata.

HATEOAS Poco Adottato

Spesso implementazione parziale riduce discoverability.

🎯 Sintesi

REST rimane lo stile dominante per API pubbliche e interne grazie a semplicità, scalabilità e aderenza agli standard web.
