🔓 OAuth 2.0

Delega di autorizzazione: accesso a risorse per conto dell'utente.

🏗️ Ruoli

Resource Owner

Utente

  • Concede accesso
  • Autentica presso IdP

Client

App richiedente

  • Usa access token
  • Non vede password (Auth Code)

Authorization Server

IdP

  • Autentica utente
  • Emette tokens

Resource Server

API protetta

  • Valida token
  • Applica scope

🔑 Grant Types Moderni

Authorization Code + PKCE

code_challenge = BASE64URL(SHA256(verifier))

Per SPA e mobile.

Client Credentials

Server-to-server senza utente.

Refresh Token

Rigenera access token.

Device Code

Dispositivi senza browser.

🧭 Flow Authorization Code (PKCE)

1

Code Challenge

Genera verifier + challenge.

2

Authorization Request

Redirect login + scope.

3

Login & Consent

Utente autorizza.

4

Authorization Code

Redirect con code.

5

Token Exchange

Code + verifier → tokens.

6

API Access

Access token nei call.

🛡️ Sicurezza

Protezione

  • PKCE obbligatorio SPA
  • State param anti-CSRF
  • Nonce per OIDC
  • HTTPS sempre

Rischi

  • Code injection
  • Token leakage redirect
  • Phishing login
  • Refresh abuso

Mitigazioni

  • SameSite cookie
  • Short access exp
  • Rotazione refresh
  • JARM / PAR (avanzato)

Errore

  • Implicit grant (deprecato)
  • Password grant (deprecato)
  • No PKCE
  • Scopes eccessivi

🔍 Scopes & Claims

Scopes

scope=profile email photos:write

Access Token

{"scope":"photos:read"}

Introspection

POST /oauth/introspect token=...

Revocation

POST /oauth/revoke token=...

🎯 Sintesi

OAuth 2.0 separa autenticazione (utente) da autorizzazione (accesso a risorse) abilitando delega controllata.

📚 Curiosità

Nato da collaborazione Twitter + Google + Yahoo (2007-2010) per superare pattern password sharing.
