🏢 Architettura Monolitica

Un'architettura monolitica è un modello di progettazione software dove tutti i componenti dell'applicazione sono unificati in un'unica unità deployabile coesa.

📜 Storia del Monolite

Dal software degli anni '60 fino ai primi web frameworks come Ruby on Rails (2004), l'architettura monolitica è stata il paradigma dominante per decenni.

1960s-1980s - Era Mainframe

Tutti i programmi erano monolitici per necessità. COBOL, FORTRAN, sistemi batch processing.

1990s-2000s - Client-Server Monoliths

Applicazioni desktop monolitiche (Microsoft Office), server applications (SAP R/3).

2004 - Rails Revolution

Ruby on Rails rende l'architettura monolitica mainstream per web applications.

A cosa serve: Sviluppare applicazioni come singola unità deployabile con codebase unificato, shared database, e deployment atomico.

Quando utilizzarlo: Applicazioni piccole-medie, team piccoli (2-8 persone), proof of concept, startup MVP, sistemi con domain boundaries poco chiari.

Come implementarlo: Single codebase, shared database, unified deployment pipeline, monolithic frameworks (Rails, Django, Spring Boot).

✅ Vantaggi

  • Semplicità sviluppo: Un solo codebase da gestire
  • Debugging facile: Stack trace complete
  • Testing integrato: End-to-end testing diretto
  • Deployment atomico: All-or-nothing deployment
  • Performance: No network latency interna

❌ Svantaggi

  • Scaling limitations: Scale tutto o niente
  • Technology lock-in: Un linguaggio/framework
  • Team constraints: Difficile parallelismo
  • Codebase complexity: Diventa ingestibile
  • Single point failure: Un bug ferma tutto

🎭 Curiosità: Il Termine "Monolite"

Il termine "monolite" deriva dall'architettura fisica: un blocco unico di pietra come gli obelischi egizi o Stonehenge. È stato coniato per l'informatica negli anni '60 per descrivere programmi "carved from a single stone" - scolpiti da una singola pietra di codice!
