🔑 JWT - JSON Web Token

Formato compatto, autoverificabile per trasportare claims tra sistemi distribuiti.

🧬 Struttura di un JWT

🔗 Formato

xxxxx.yyyyy.zzzzz
header.payload.signature

Tre blocchi Base64URL separati da punti.

🧾 Header

{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "key-2025-01"
}

Algoritmo, tipo token, key id.

📜 Payload (Claims)

{
  "sub": "user-123",
  "role": "admin",
  "scope": ["users:read"],
  "iat": 1732465200,
  "exp": 1732468800
}

Dati business + claims standard.

✍️ Signature

sign( base64url(header)+"."+base64url(payload), key )

Garantisce integrità & autenticità.

📑 Claims Principali

Identità

  • sub: Subject (ID utente)
  • iss: Issuer
  • aud: Audience
  • jti: ID token unico

Tempo

  • iat: Issued At
  • exp: Expiry
  • nbf: Not Before
  • auth_time: Tempo autenticazione

Autorizzazione

  • scope: Permessi granulari
  • role: Ruolo
  • permissions: Lista custom
  • acl: Access list

Rischi

  • Token leakage
  • No revoca nativa
  • Alg confusion
  • XSS exposure

🔐 Firmare & Verificare

HS256

jwt.sign(payload, secret, { expiresIn: '15m' });
jwt.verify(token, secret);

Chiave simmetrica condivisa.

RS256

jwt.sign(payload, privateKey, { algorithm: 'RS256' });
jwt.verify(token, publicKey);

Chiave privata/pubblica – ideale microservizi.

JWKS

GET /.well-known/jwks.json
{ "keys": [ {"kid":"2025-01","kty":"RSA", ...} ] }

Rotazione & discovery chiavi.

Alg None (storico)

{ "alg": "none" } // Vulnerabilità

Whitelist algoritmi sempre.

🔄 Access vs Refresh

⚡ Access Token

Breve durata (5–15m)

  • Usato nelle API
  • Nessuna revoca centralizzata
  • Contiene scopes
  • Lightweight

🔁 Refresh Token

Durata lunga

  • Conservato HttpOnly cookie
  • Rotazione obbligatoria
  • Revocabile
  • Tracciato server-side

🧰 Middleware di Validazione

Express

function auth(req,res,next){
  const t=req.headers.authorization?.split(' ')[1];
  try {
    req.user = jwt.verify(t, publicKey, {
      algorithms:['RS256'],
      issuer:'auth.example.com',
      audience:'api.example.com'
    });
    next();
  } catch { res.status(401).json({error:'invalid_token'}); }
}

Rotazione Refresh

POST /auth/refresh
1. Verifica refresh
2. Invalida jti precedente
3. Emissione nuova coppia
4. Log sicurezza

Logout Sicuro

POST /auth/logout
blacklist.add(refresh.jti)
clear refresh cookie

Blacklist

// Redis TTL = exp - now
SET blacklist:<jti> 1 EX 900

🛡️ Best Practices

Sicurezza

  • Exp breve access token
  • Refresh HttpOnly
  • Validate iss/aud/exp/nbf
  • Rotate keys (kid)

Anti-Abuso

  • Rate limiting
  • Device binding opzionale
  • Anomaly detection
  • Blacklist compromise

Operatività

  • Monitor exp near-future
  • Central JWKS cache
  • Structured logging claims
  • Tracing correlation (jti)

Rischi Comune

  • Salvare in localStorage
  • Exp troppo lunghi
  • Algoritmo non fissato
  • Claims eccessivi

🎯 Sintesi

JWT abilita autenticazione stateless scalabile. Minimizza durata, proteggi refresh, monitora anomalie.

🧪 Curiosità

Il bug "alg none" ha spinto audit globali migliorando la robustezza delle librerie.
